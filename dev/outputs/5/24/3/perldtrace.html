NAME

perldtrace - Perl's support for DTrace

SYNOPSIS

  1.  # dtrace -Zn 'perl::sub-entry, perl::sub-return { trace(copyinstr(arg0)) }'
  2.  dtrace: description 'perl::sub-entry, perl::sub-return ' matched 10 probes
  4.  # perl -E 'sub outer { inner(@_) } sub inner { say shift } outer("hello")'
  5.  hello
  7.  (dtrace output)
  8.  CPU     ID                    FUNCTION:NAME
  9.    0  75915       Perl_pp_entersub:sub-entry   BEGIN
  10.    0  75915       Perl_pp_entersub:sub-entry   import
  11.    0  75922      Perl_pp_leavesub:sub-return   import
  12.    0  75922      Perl_pp_leavesub:sub-return   BEGIN
  13.    0  75915       Perl_pp_entersub:sub-entry   outer
  14.    0  75915       Perl_pp_entersub:sub-entry   inner
  15.    0  75922      Perl_pp_leavesub:sub-return   inner
  16.    0  75922      Perl_pp_leavesub:sub-return   outer

DESCRIPTION

DTrace is a framework for comprehensive system- and application-level tracing. Perl is a DTrace provider, meaning it exposes several probes for instrumentation. You can use these in conjunction with kernel-level probes, as well as probes from other providers such as MySQL, in order to diagnose software defects, or even just your application's bottlenecks.

Perl must be compiled with the http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"> http://www.w3.org/1999/xhtml"> perltidy

perltidy

-Dusedtrace option in order to make use of the provided probes. While DTrace aims to have no overhead when its instrumentation is not active, Perl's support itself cannot uphold that guarantee, so it is built without DTrace probes under most systems. One notable exception is that Mac OS X ships a /usr/bin/perl with DTrace support enabled.

PROBES

  • sub-entry(SUBNAME, FILE, LINE, PACKAGE)

    Traces the entry of any subroutine. Note that all of the variables refer to the subroutine that is being invoked; there is currently no way to get ahold of any information about the subroutine's caller from a DTrace action.

    http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"> http://www.w3.org/1999/xhtml"> perltidy

    perltidy

     :*perl*::sub-entry {
     printf("%s::%s entered at %s line %d\n",
           copyinstr(arg3), copyinstr(arg0), copyinstr(arg1), arg2);
 }
  • sub-return(SUBNAME, FILE, LINE, PACKAGE)

    Traces the exit of any subroutine. Note that all of the variables refer to the subroutine that is returning; there is currently no way to get ahold of any information about the subroutine's caller from a DTrace action.

    http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"> http://www.w3.org/1999/xhtml"> perltidy

    perltidy

     :*perl*::sub-return {
     printf("%s::%s returned at %s line %d\n",
           copyinstr(arg3), copyinstr(arg0), copyinstr(arg1), arg2);
 }

  • op-entry(OPNAME)

    Traces the execution of each opcode in the Perl runloop. This probe is fired before the opcode is executed. When the Perl debugger is enabled, the DTrace probe is fired after the debugger hooks (but still before the opcode itself is executed).

    http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"> http://www.w3.org/1999/xhtml"> perltidy

    perltidy

     :*perl*::op-entry {
     printf("About to execute opcode %s\n", copyinstr(arg0));
 }
  • loading-file(FILENAME)

    Fires when Perl is about to load an individual file, whether from use, require, or do. This probe fires before the file is read from disk. The filename argument is converted to local filesystem paths instead of providing http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"> http://www.w3.org/1999/xhtml"> perltidy

    3.  }
  • loaded-file(FILENAME)

    Fires when Perl has successfully loaded an individual file, whether from use, require, or do. This probe fires after the file is read from disk and its contents evaluated. The filename argument is converted to local filesystem paths instead of providing http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"> http://www.w3.org/1999/xhtml"> perltidy
