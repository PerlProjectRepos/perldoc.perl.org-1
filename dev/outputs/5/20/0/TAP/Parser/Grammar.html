NAME

TAP::Parser::Grammar - A grammar for the Test Anything Protocol.

VERSION

Version 3.30

SYNOPSIS

  use TAP::Parser::Grammar;

    iterator => $tap_parser_iterator,
    parser   => $tap_parser,
    version  => 12,
  });

  my $result = $grammar->tokenize;

DESCRIPTION

http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd"> http://www.w3.org/1999/xhtml"> perltidy

perltidy

TAP::Parser::Grammar tokenizes lines from a TAP::Parser::Iterator and constructs TAP::Parser::Result subclasses to represent the tokens.

Do not attempt to use this class directly. It won't make sense. It's mainly here to ensure that we will be able to have pluggable grammars when TAP is expanded at some future date (plus, this stuff was really cluttering the parser).

 my @tokens;
 my $grammar = TAP::Grammar->new;
 LINE: while ( defined( my $line = $parser->_next_chunk_of_tap ) ) {
     for my $type ( $grammar->token_types ) {
         my $syntax  = $grammar->syntax_for($type);
         if ( $line =~ $syntax ) {
             my $handler = $grammar->handler_for($type);
             push @tokens => $grammar->$handler($line);
             next LINE;
         }
     }
     push @tokens => $grammar->_make_unknown_token($line);
 }

TAP GRAMMAR

NOTE: This grammar is slightly out of date. There's still some discussion about it and a new one will be provided when we have things better defined.

The TAP::Parser does not use a formal grammar because TAP is essentially a stream-based protocol. In fact, it's quite legal to have an infinite stream. For the same reason that we don't apply regexes to streams, we're not using a formal grammar here. Instead, we parse the TAP in lines.

For purposes for forward compatibility, any result which does not match the following grammar is currently referred to as TAP::Parser::Result::Unknown. It is not a parse error.

A formal grammar would look similar to the following:

  1.  (*
  2.      For the time being, I'm cheating on the EBNF by allowing
  3.      certain terms to be defined by POSIX character classes by
  4.      using the following syntax:
  6.        digit ::= [:digit:]
  8.      As far as I am aware, that's not valid EBNF.  Sue me.  I
  9.      didn't know how to write "char" otherwise (Unicode issues).
  10.      Suggestions welcome.
  11.  *)
  13.  tap            ::= version? { comment | unknown } leading_plan lines
  14.                     |
  15.                     lines trailing_plan {comment}
  17.  version        ::= 'TAP version ' positiveInteger {positiveInteger} "\n"
  19.  leading_plan   ::= plan skip_directive? "\n"
  21.  trailing_plan  ::= plan "\n"
  23.  plan           ::= '1..' nonNegativeInteger
  25.  lines          ::= line {line}
  27.  line           ::= (comment | test | unknown | bailout ) "\n"
  29.  test           ::= status positiveInteger? description? directive?
  31.  status         ::= 'not '? 'ok '
  33.  description    ::= (character - (digit | '#')) {character - '#'}
  35.  directive      ::= todo_directive | skip_directive
  37.  todo_directive ::= hash_mark 'TODO' ' ' {character}
  39.  skip_directive ::= hash_mark 'SKIP' ' ' {character}
  41.  comment        ::= hash_mark {character}
  43.  hash_mark      ::= '#' {' '}
  45.  bailout        ::= 'Bail out!' {character}
  47.  unknown        ::= { (character - "\n") }
  49.  (* POSIX character classes and other terminals *)
  51.  digit              ::= [:digit:]
  52.  character          ::= ([:print:] - "\n")
  53.  positiveInteger    ::= ( digit - '0' ) {digit}
  54.  nonNegativeInteger ::= digit {digit}
